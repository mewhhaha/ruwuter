# Static-route fast path

**Status:** implemented · **Size:** S · **Files:** `src/router.ts` (compile step only)

## Problem

`Router.handle` matches by scanning every compiled route and calling `URLPattern.exec` until one
hits (`src/router.ts:424-431`), and `fragmentResponse` runs its own scan first on every request. For
an app with mostly static paths (`/`, `/about`, `/pricing`, ...), each request pays N pattern
executions for what could be one Map lookup. `URLPattern.exec` is not cheap in workerd.

This is invisible at 10 routes and real at 200.

## Proposal

All patterns are known at `Router()` time (they come from the generated route table), so the compile
step in `Router()` can bucket them:

- A pattern string with no `:param`, `*`, or `(...)` group is static →
  `staticRoutes:
  Map<pathname, CompiledRoute>`.
- Everything else stays in the ordered scan array.

`handle` tries `staticRoutes.get(url.pathname)` first, falls back to the scan. Specificity sorting
is unaffected because a static pattern that exists in the map would also have won the scan (exact
match beats params in `bySpecificity`, and duplicate patterns are already a generation-time error).

~20 lines, zero API change, no new state per request.

## Only if it ever matters

The scan itself could become a segment trie generated by `fs-routes` (the generator already sorts by
specificity and knows the grammar). That's real complexity for a case — hundreds of _dynamic_ routes
— that may never exist. Measure first: add a micro-benchmark (`URLPattern.exec` × route-count in
workerd via `wrangler dev --test-scheduled` or a plain Deno bench as a proxy) and let numbers decide
whether anything beyond the Map is warranted.

## Implementation

Router compilation buckets static paths into a Map and dynamic routes into the ordered fallback. The
matcher still preserves the public route array's first-match behavior, duplicate-static order, and
fragment fallback across matching routes. Specificity-sorted generated tables take the O(1) path
without executing `URLPattern`.
